# Stripe Service

Thin Stripe wrapper. Mirrors a subset of Stripe API objects (customers, checkout
sessions, payment intents, billing portal sessions) under Stripe-native routes
and DB tables. Internal-only HTTP API consumed by other Distribute services
(billing-service, future subscription-service). The service has no business
knowledge of its callers — all reload/dedupe logic lives in clients.

## Commands
- `npm run dev` — Start dev server with nodemon
- `npm run build` — Compile TypeScript + generate OpenAPI spec
- `npm test` — Run all tests
- `npm run test:unit` — Unit tests only
- `npm run test:integration` — Integration tests only
- `npm run db:generate` — Generate Drizzle migrations
- **Merge flow:** GitHub auto-merge is DISABLED on this repo (`gh pr merge --auto` errors `Auto merge is not allowed`). Wait for the `test` CI check to go green (`gh pr checks <PR> --watch`), then merge manually (`gh pr merge <PR> --squash --delete-branch`). Bugfix/feature PRs base `staging`; promote to `main` is a separate step.
- `npm run db:push` — Push schema to database

## Architecture
- **src/schemas.ts** — Zod schemas + OpenAPI path registration (single source of truth)
- **src/middleware/serviceAuth.ts** — X-API-Key authentication
- **src/middleware/identityHeaders.ts** — Requires x-org-id + x-user-id; logs optional context headers
- **src/middleware/callLog.ts** — Appends to api_call_log on every authenticated request
- **src/lib/stripe-client.ts** — Per-org Stripe SDK factory + webhook verifier
- **src/lib/key-client.ts** — Key-service HTTP client
- **src/lib/resolve-stripe-key.ts** — Resolves Stripe key from key-service via orgId + userId
- **src/lib/request-context.ts** — Builds per-request Stripe client + identity context
- **src/lib/event-processor.ts** — Bronze (events) ingestion + silver (customers/payment_intents/checkout_sessions) projection. `processEvent()` inserts webhook/poll events into bronze, then calls `projectSilverFromBronze(objectId, orgId)` which reads `MAX(events.created_stripe)` for that object_id and upserts silver from the latest payload. `recordApiSnapshot(obj, kind, orgId)` is the entry point for API-response paths — inserts a synthetic `api_snapshot.<kind>` event into bronze, then projects. `repairAllSilverFromBronze()` runs at boot to re-project every distinct object_id (idempotent, bounded by event-count). Also dispatches post-mirror side-effects (default-PM promotion, Stripe-fee declaration) for real Stripe events only — synthetic events never trigger side-effects.
- **src/lib/promote-default-pm.ts** — Side-effect fired on `checkout.session.completed` and `setup_intent.succeeded`: if the customer has no `invoice_settings.default_payment_method` yet, promote the PM used in the session/intent so off_session reloads + `has_payment_method` work going forward. Idempotent.
- **src/lib/declare-fees.ts** — Side-effect fired on `charge.succeeded`, `charge.refunded`, `charge.dispute.created`, `payout.failed`: looks up the Stripe `balance_transaction.fee`, resolves the org via the customers mirror, reads the org's keySource from key-service, then declares the fee as a cost on a synthetic platform-run in runs-service. Idempotent per Stripe `balance_transaction.id` via the runs-service `idempotencyKey` field.
- **src/lib/runs-client.ts** — HTTP client for runs-service. `createPlatformRun`, `addPlatformRunCost`, `updatePlatformRunStatus`. Used exclusively by the fee declaration side-effect.
- **src/lib/event-poller.ts** — 5-min background poll of `GET /v1/events` from Stripe (webhook-loss recovery)
- **src/lib/historical-backfill.ts** — Boot-time full-history back-fill of every mirrored table (customers, payment_intents, checkout_sessions, customer_balance_transactions) via Stripe object-list APIs (events.list useless beyond 30-day retention)
- **src/routes/** — `customers.ts`, `internal.ts` (all `/internal/*` server-to-server routes: org teardown + user-less org-scoped balance reads), `checkout-sessions.ts`, `payment-intents.ts`, `payment-methods.ts`, `billing-portal-sessions.ts`, `public-stats.ts`, `webhooks.ts`, `health.ts`
- **src/db/schema.ts** — Drizzle table definitions (Stripe-shape mirror)
- **scripts/generate-openapi.ts** — OpenAPI spec generator

## Routes

`/v1/*` = auth'd (X-API-Key + identity headers). `/internal/*` = auth'd (X-API-Key only, NO identity headers — server-to-server, org keyed off the path). `/public/*` = no auth, cross-org. `/health`, `/openapi.json` = public utility.

```
POST   /v1/customers                       — create (idempotent per org; attaches user email+name from client-service)
GET    /v1/customers/:id                   — retrieve (DB-first, Stripe fallback)
POST   /v1/customers/:id                   — update
GET    /v1/customers                       — list (DB-only)
DELETE /internal/customers/by-org/:orgId   — org teardown: delete Stripe customer online + tombstone mirror (idempotent)
GET    /internal/customers/by-org/:orgId        — user-less: org's customer raw_json (DB mirror), 404 if none
GET    /internal/payment_intents/by-org/:orgId  — user-less: org's PIs as Stripe list (DB mirror, no limit)
GET    /internal/payment_methods/by-org/:orgId  — user-less: org customer's PMs (live Stripe, platform key); ?type=card
POST  /v1/checkout/sessions               — create (line_items required for mode payment/subscription; forbidden for mode setup)
GET   /v1/checkout/sessions/:id           — retrieve (DB-first, Stripe fallback)
GET   /v1/checkout/sessions               — list (DB-only)
POST  /v1/payment_intents                 — create
GET   /v1/payment_intents/:id             — retrieve (DB-first, Stripe fallback)
GET   /v1/payment_intents                 — list (DB-only) — used by callers to check in-flight reloads
GET   /v1/payment_methods?customer=&type= — live Stripe passthrough (no mirror); 404s when customer not in caller's org
POST  /v1/billing_portal/sessions         — create (no DB persistence; single-use URL)
POST  /v1/webhooks                        — Stripe webhook handler (signature-verified)
GET   /public/stats/billing               — cross-org aggregate Stripe stats (no auth)
```

## Key patterns

- **Bronze/silver — silver is projected from bronze, never written directly.** The `events` table is bronze (append-only, full Stripe-event payload as `jsonb`, idempotent insert via `ON CONFLICT (id) DO NOTHING`). The `customers`, `payment_intents`, and `checkout_sessions` tables are silver — typed projections of bronze. The contract: any code that needs to write silver MUST go through `projectSilverFromBronze(objectId, orgId)` (which reads `MAX(events.created_stripe)` per object_id) or `recordApiSnapshot(obj, kind, orgId)` (which inserts a synthetic `api_snapshot.<kind>` bronze event with `created_stripe = floor(Date.now()/1000)` then projects). The `upsertCustomer` / `upsertPaymentIntent` / `upsertCheckoutSession` helpers in `event-processor.ts` are exported only for tests — production callers using them directly bypass the race guarantee. **Why:** Stripe webhook delivery order is not guaranteed. Pre-projection, `ON CONFLICT DO UPDATE` was last-write-wins, so an older `payment_intent.created` snapshot arriving after a newer `payment_intent.succeeded` would clobber silver back to `requires_payment_method`. This silently broke billing-service's `sumSucceededTopupsForCustomer` and triggered runaway reload cascades (DIS-70). Projection ordered by `events.created_stripe DESC, received_at DESC` eliminates the race by construction — Stripe `event.created` is monotonic per object across state transitions. Synthetic api_snapshot events use server time, so they dominate any past webhook for the same object; later real webhooks with higher `event.created` win automatically.
- **Stripe-shape passthrough.** Request bodies are validated with `.passthrough()` Zod schemas and forwarded verbatim to the Stripe SDK. Responses are the unmodified Stripe object. No custom request/response naming. **If a Stripe object does NOT have a corresponding Stripe API endpoint, do NOT invent a convenience wrapper here** — direct the caller to read the field off the parent Stripe-API endpoint we already mirror (e.g. `customer.balance` and `customer.invoice_settings.default_payment_method` read from `GET /v1/customers/:id`, not from `/v1/customers/balance` or `/v1/customers/has-payment-method`). Convenience flatteners with snake_case wrappers (`balance_cents` as string, flattened status enums, etc.) violate this rule.
- **1:1 org ↔ Stripe customer.** Each org has exactly one Stripe customer. Org-implicit reads (e.g. `GET /v1/balance_transactions`) resolve `customers WHERE org_id = $1 ORDER BY created_stripe DESC LIMIT 1`; 404 if no customer. Callers (billing-service) do NOT store `cus_xxx` — stripe-service is the source of truth for the mapping.
- **`POST /v1/customers` is idempotent per org — enforces the 1:1 invariant server-side.** Before creating, it resolves the org's existing mirrored customer (`customers WHERE org_id = $1 ORDER BY synced_at DESC LIMIT 1`); if one exists it returns it (no Stripe create). This kills duplicate Stripe customers from caller retries / races / multiple account-setup paths — billing's own `billing_accounts` guard was the only protection before, so any hiccup there minted a second `cus_…`. The create is now retry-safe, which is what makes the fail-loud email resolution below acceptable.
- **`POST /v1/customers` attaches the user's email + name from client-service.** stripe-service has no email of its own (same as it has no Stripe key of its own — resolved from key-service). Callers (billing-service `ensureCustomer`) POST an **empty body** `{}`, so the Stripe customer was born with no email. The POST now resolves `{email, firstName, lastName}` from client-service `GET /internal/users/:userId` (keyed off the `x-user-id` it already receives) via `src/lib/client-service-client.ts`, and stamps them — **only when the caller didn't supply an email** (caller-supplied values win; callers that already have an email skip the client-service hop). Existing emailless customers are **backfilled on the idempotent-return path** (`customers.update` with the resolved email). **Fail-loud:** an infra failure (network/5xx) propagates → 5xx → caller retries (safe, the create is idempotent). A 404 (no such user on record) is NOT a failure — it means there's genuinely no email to attach, so the customer is created without one rather than blocking the org's billing setup. Keeping email-resolution server-side fixes it transparently for every caller (billing today, future subscription-service) instead of each caller re-implementing it.
- **Public routes.** `/public/*` bypasses `serviceAuth` and `requireIdentityHeaders` middleware. Reserved for cross-org domain aggregates (stats, status pages). Not for Stripe-shape endpoints — those live under `/v1/`. Adding a new `/public/*` route is a deliberate architectural choice; reject the request unless the data is genuinely cross-org and non-sensitive.
- **Internal routes.** `/internal/*` keeps `serviceAuth` (X-API-Key mandatory) but is exempt from `requireIdentityHeaders` (the exemption is in `identityHeaders.ts`, NOT in `serviceAuth.ts` — never weaken the key check). Reserved for server-to-server platform operations where the caller has no end-user identity, the org is keyed off the path/body, and Stripe access (when needed) uses the **platform key** (`getPlatformStripe()`, single-account model) — there is no end-user to resolve a per-org key against. Examples: org teardown (`DELETE …/by-org/:orgId`) and the user-less org-scoped balance reads (`GET …/by-org/:orgId` for customers / payment_intents / payment_methods). **Belongs under `/v1/` instead, NOT `/internal/`:** any Stripe-shape read/write that needs **per-org, per-user key resolution** (`resolveStripeKey(orgId, userId)` — BYOK vs platform chosen per the user's org) or that stamps end-user identity. The discriminator is the KEY-RESOLUTION path (platform vs per-user), not "is it Stripe-shape". The user-less balance reads exist because billing-service's affordability + dunning schedulers are machine-triggered (no end-user) and must compose an org's balance with only X-API-Key + org (no fake/sentinel x-user-id) — see next bullet.
- **Org teardown — `DELETE /internal/customers/by-org/:orgId`.** Part of the platform org cascade-teardown (api-service `DELETE /internal/admin/orgs/:orgId` orchestrates; this is its Stripe leg). Resolves `customers WHERE org_id = $1`, deletes each at Stripe via `stripe.customers.del(id)` using the **platform key** (`getPlatformStripe()` — single-account model, same as poller/back-fill/side-effects; per-org `resolveStripeKey` needs a userId teardown doesn't have), then **durably tombstones** silver. **Tombstone, never raw `DELETE FROM customers`:** a raw silver delete is resurrected on the next boot by `repairAllSilverFromBronze()`, which re-projects every `object_id` from the latest bronze event (a leftover `customer.created` / `api_snapshot.customer` event re-upserts the row). Instead it calls `recordApiSnapshot({ id, deleted: true, livemode }, "customer", orgId)` — the synthetic event's `created_stripe = now` dominates the projection's `ORDER BY created_stripe DESC`, so `projectSilverFromBronze` takes the `DeletedCustomer` branch (`db.delete(customers)`) and silver stays deleted across every re-projection; a later real `customer.deleted` webhook agrees. **Idempotent:** no customer row → 200, nothing deleted (no Stripe call). `resource_missing` from Stripe (already gone) is success and still tombstones. **Fail loud:** any other Stripe error propagates → non-2xx → caller retries. Order is delete-Stripe-then-tombstone so a crash mid-op re-runs cleanly. Scope is the customer only — PIs/checkout-sessions silver are NOT purged (they're historical records, not live billing entities, and have no `deleted` projection branch to tombstone against).
- **User-less org-scoped balance reads — `GET /internal/{customers,payment_intents,payment_methods}/by-org/:orgId`.** Back billing-service's balance composition (`computeBalance`) when the caller is a machine with no end-user — campaign-service's credit-affordability gate and the dunning scheduler are both machine-triggered. These mirror the `/v1` reads (`getCustomerByOrg` ← `GET /v1/customers?limit=1`; `sumSucceededTopupsForCustomer` ← `GET /v1/payment_intents`; `hasAttachedCardPm` ← `GET /v1/payment_methods`) in **passthrough Stripe shape** but key the org off the path and require **no x-user-id**. customers + payment_intents are pure DB-mirror reads (no Stripe call, no key). payment_methods is a live Stripe `paymentMethods.list` via the **platform key** (`getPlatformStripe()`, same single-account model as teardown — no user to resolve a per-org key against), customer resolved from the mirror (404 if none). payment_intents returns ALL the org's PIs with **no limit** (caller sums succeeded top-ups across the full set). **Why this surface exists:** stripe-service's global `requireIdentityHeaders` hard-requires x-user-id on every `/v1/*` route (Stripe key is resolved per-org-per-user there); billing was working around it by injecting a **zero-uuid x-user-id sentinel**, which the new affordability handler forgot → `400 Missing required header: x-user-id` → 502 → affordability fails OPEN (storm-protection inert). The fix is this user-less surface + billing dropping the sentinel on its balance path — NOT a fake user-id (stripe-service#77). No service-side balance composition — stripe-service has no business knowledge of callers; billing owns the `topups − usage` math.
- **org_id stamping.** On every `POST /v1/{customers,checkout/sessions,payment_intents}`, stripe-service stamps `metadata.org_id = <x-org-id>` so webhook events route back to the right tenant via `event.data.object.metadata.org_id`.
- **Webhook + 5-min poll sync.** Webhooks are primary. A background `setInterval(5min)` pulls Stripe `events.list` since the cursor stored in `event_sync_cursor` and processes any missed events. Both paths share `processEvent()` which is idempotent via `ON CONFLICT DO NOTHING` on `events.id`.
- **Default-PM auto-promotion.** Stripe Checkout attaches the chosen PM to the customer but does NOT set it as `invoice_settings.default_payment_method`. Without a default, billing-service `deriveHasPaymentMethod` returns false (dashboard "No card") and off_session PaymentIntents fail with `requires_payment_method`. `processEvent()` calls `promoteDefaultPaymentMethod(event, stripe)` after the mirror upsert for `checkout.session.completed` (modes `payment`/`setup`) and `setup_intent.succeeded`. It retrieves the customer; if a default PM is already set, it's a no-op. Otherwise it derives the PM (PI/SI lookup for checkout, payload for SI), calls `customers.update(id, { invoice_settings: { default_payment_method: pm } })`, and re-mirrors via `upsertCustomer`. Errors propagate so the webhook returns 5xx and Stripe retries. Uses the platform Stripe key (same single-account model as the poller + back-fill). Forward-only — no historical back-fill of older customers. **Note:** the promoted PM is whatever the customer used to complete checkout, which may be a Stripe Link / wallet PM. Stripe refuses to charge Link / wallet PMs in off_session mode via the `invoice_settings.default_payment_method` fallback ("You cannot confirm this PaymentIntent because it's missing a payment method"). Callers that need off_session must NOT rely on the default-PM fallback — instead list the customer's PMs via `GET /v1/payment_methods?customer=&type=card` and pass `payment_method` explicit on the PI create call.
- **`GET /v1/payment_methods` is a live Stripe passthrough.** No mirror table. Calls `stripe.paymentMethods.list({ customer, type? })` after verifying the customer exists in the caller's org mirror (prevents cross-org PM enumeration via guessed cus_xxx IDs). Live-Stripe (not mirror) because PMs detach / attach without always emitting `customer.updated` events, so any local cache would drift. Callers that need an off_session-capable card PM should query this endpoint and pick the first card-type result rather than reading `customer.invoice_settings.default_payment_method` (which may be a Link / wallet PM Stripe refuses for off_session).
- **Stripe-fee declaration to runs-service.** On `charge.succeeded` / `charge.refunded` / `charge.dispute.created` / `payout.failed`, `processEvent()` calls `declareFeesForEvent(event, stripe)` after the mirror upsert. For each fee-bearing balance-transaction referenced by the event, it retrieves the BT (via the platform Stripe key), resolves the owning org via `customers.org_id`, reads `keySource` from key-service `GET /keys/stripe/source`, then POSTs to runs-service: `POST /v1/platform-runs` (taskName = the event type, x-org-id header set when an org is attributable), `POST /v1/platform-runs/:id/costs` (`costName` ∈ {`stripe-processing-fee`, `stripe-refund-fee`, `stripe-dispute-fee`, `stripe-payout-failure-fee`}, `costSource` from keySource for org-attributable events, `platform` for `payout.failed` and customer-less charges, `quantity` = `balance_transaction.fee` in cents, `status: actual`), then `PATCH /v1/platform-runs/:id` to `completed`. The Stripe `balance_transaction.id` is passed as `idempotencyKey` on both the run and the cost so Stripe webhook redelivery is safe (runs-service returns the existing rows on the second call). Errors propagate so the webhook returns 5xx and Stripe retries; on cost-write failure the run is best-effort patched to `failed` before re-throw. **costName must match the kebab-case strings registered in costs-service** — `stripe.processing_fee` etc. will 422 with "Unknown cost name". The 2× cost-risk-multiplier applied by costs-service is intentional platform-wide convention; not a stripe-service concern. Fee amount is read from `balance_transaction.fee` so non-charge events (refund/payout/dispute) only declare when Stripe populates the fee field (best-effort capture for those classes).
- **Boot-time historical back-fill.** On every boot, `backfillHistorical()` runs as a fire-and-forget background task fired right after `app.listen()`. Iterates `customers.list`, `paymentIntents.list`, and `checkout.sessions.list` globally. Routes each object to its tenant via `metadata.org_id`, then falls back to the local `customers` mirror's `org_id` for PaymentIntents and CheckoutSessions whose own metadata lacks `org_id` (PIs created via Checkout sessions don't inherit the session's metadata, so the customer lookup is the only recoverable source). Same `resolveOrgId(metadataOrgId, customerId)` helper in `event-processor.ts` is used by webhook + poll-driven upserts so the bug class cannot recur. Only stamps `"unknown"` when both sources are empty. Idempotent via `ON CONFLICT DO UPDATE` so re-runs refresh `raw_json` / status. Failure is logged via `console.error` but does NOT crash the service — the port is bound first so callers never get ECONNREFUSED during the back-fill window. Recovers Feb→Apr 2026 history that the 30-day events.list cursor cannot reach.
- **`customer.balance` is polluted — stripped on write.** Pre-#104 (billing-service), every usage tick wrote a `usage_applied` CBT into Stripe via `stripe.customers.createBalanceTransaction(...)`. Stripe's `customer.balance` is the running sum of those CBTs plus real top-up credits, so it no longer represents a pure prepaid balance for any org with pre-#104 history. `upsertCustomer()` strips `raw_json.balance` before persisting; no stripe-service response, mirror table, or back-fill exposes the field. Downstream consumers must compute credit balance from `payment_intents.amount_received` + their own promo ledger, NOT from `customer.balance`.
- **No `customer_balance_transactions` exposure.** Mirror table + `GET /v1/balance_transactions` route + webhook handler for `customer_balance_transaction.*` events are all removed. Same legacy-pollution rationale: individual CBTs cannot be cleanly partitioned into "real credit" vs "legacy usage_applied" without unreliable type heuristics. Drop the surface entirely. If a downstream service needs credit history, source it from its own ledger.
- **No service-side reload dedupe.** Stripe-service has no in-flight lock, no customer-scope coalesce, no gap-fill logic. Clients (billing-service) inspect in-flight PaymentIntents via `GET /v1/payment_intents?customer=cus_X` and own the business logic for "don't trigger reload too often".
- **Idempotency-Key forwarding.** If the caller sends an `Idempotency-Key` header, it's forwarded verbatim to the Stripe SDK call. Otherwise the SDK is called without one.
- **Reads.** Single-object GET = DB-first + Stripe fallback when row missing. List = DB-only (no live Stripe fetch). Webhook + poll keep DB fresh.
- **Auth.** `X-API-Key` mandatory on all non-webhook, non-public routes (including `/internal/*`). `x-org-id` + `x-user-id` mandatory on `/v1/*` because Stripe key resolution is per-org; `/internal/*` is exempt (keys the org off the path). `x-brand-id`, `x-campaign-id`, `x-workflow-slug` optional — logged to `api_call_log` when present.
- **Multi-tenant Stripe key.** `resolveStripeKey(orgId, userId)` calls key-service `GET /keys/stripe/decrypt`. Per-request Stripe client (`makeStripeClient(key)`), no shared SDK instance.

## DB tables

```
customers                       — id (cus_…)   PK, org_id, email, name, metadata, raw_json, synced_at
checkout_sessions               — id (cs_…)    PK, org_id, customer, payment_intent, mode, status, raw_json, synced_at
payment_intents                 — id (pi_…)    PK, org_id, customer, amount, status, raw_json, synced_at
events                          — id (evt_…)   PK, type, object_id, payload, source ('webhook'|'poll'), received_at
event_sync_cursor               — id=1, last_event_id, last_synced_at  (single-row)
api_call_log                    — request audit: method, path, status, identity headers, stripe_object_id, duration_ms
```

`raw_json` holds the unmodified Stripe object — `GET /v1/{obj}/:id` returns this directly so the response shape matches Stripe verbatim. Exception: `customers.raw_json.balance` is stripped on every write (see "Polluted `customer.balance`" key pattern).

## Environment variables
- `STRIPE_SERVICE_DATABASE_URL` — Neon Postgres connection string
- `STRIPE_SERVICE_API_KEY` — Service-to-service auth key
- `KEY_SERVICE_URL` / `KEY_SERVICE_API_KEY` — Key service connection. Resolves three Stripe-related keys at runtime via key-service: per-org Stripe keys (`/keys/stripe/decrypt`), the platform Stripe key for the event poller (`/keys/platform/stripe/decrypt`), and the Stripe webhook signing secret (`/keys/platform/stripe-webhook/decrypt`). Also reads org `keySource` preference via `/keys/stripe/source` during webhook fee declaration. No Stripe secret is ever stored in env vars on this service.
- `RUNS_SERVICE_URL` / `RUNS_SERVICE_API_KEY` — Runs-service connection for Stripe-fee declaration on webhook events. See "Stripe-fee declaration to runs-service" key pattern.
- `CLIENT_SERVICE_URL` / `CLIENT_SERVICE_API_KEY` — Client-service connection. Resolves the end-user's email + name from `x-user-id` (`GET /internal/users/:userId`) so `POST /v1/customers` stamps an email on the Stripe customer. See "`POST /v1/customers` attaches the user's email + name" key pattern.
- `RUN_EVENT_POLLER` — set to `"false"` to disable the background poller (default: enabled)
- `PORT` — Server port (default 3011)

## Out of scope (Phase 2)

Products, prices, subscriptions, invoices (Stripe objects not currently used by Distribute callers). Refund and dispute Stripe objects are not mirrored as tables, but their fees ARE declared to runs-service via the webhook fee-declaration side-effect (see key patterns). Push-style webhook fan-out to downstream services. Reload-specific business endpoints (those live in billing-service). `customer.balance` and `customer_balance_transactions` (intentionally removed, see key patterns).
